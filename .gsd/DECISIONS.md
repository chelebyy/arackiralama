# Decisions

<!-- Append-only register of architectural and pattern decisions -->

| ID | Decision | Rationale | Date |
|----|----------|-----------|------|
| D001 | Modular Monolith | Microservice-ready, low operational complexity | 2026-03-14 |
| D002 | PostgreSQL + Redis | ACID + Cache combination | 2026-03-14 |
| D003 | Iyzico payment | Local provider for TR market | 2026-03-14 |
| D004 | Principal entities auto-normalize email on assignment | Keeps `Email`/`NormalizedEmail` invariant at the domain boundary and reduces call-site drift before query/index hardening tasks | 2026-03-15 |
| D005 | Auth persistence uses shared principal-discriminated token records (`AuthSession`, `PasswordResetToken`) with hash-at-rest fields only | Keeps admin/customer principals separate while enabling shared revocation/reset flows and enforces token-at-rest redaction at the schema level | 2026-03-15 |
| D006 | EF auth persistence maps `AuthPrincipalType` as string and enforces unique `normalized_email` indexes for both principals | Keeps schema diagnostics human-readable (`principal_type`) while hardening case-insensitive identity uniqueness at the database layer | 2026-03-15 |
| D007 | Auth foundation migration backfills normalized emails and fails fast on duplicate normalized principals before unique-index enforcement | Makes rollout safe for guest-created customer history by surfacing deterministic precondition errors instead of silently writing invalid defaults or failing late at index creation | 2026-03-15 |
| D008 | JWT issuance now uses a shared principal claim model (`JwtPrincipalClaims`) for admin/customer tokens with mandatory `sid` and `ver` claims and a 15-minute access-token lifetime | Establishes a single token contract for both principal types ahead of session validation, while preserving Admin/SuperAdmin role-policy compatibility and making session diagnostics claim-decodable | 2026-03-15 |
| D009 | Refresh tokens are issued as 64-byte CSPRNG Base64Url values with `sha256:`-prefixed hash-at-rest and constant-time verification helper methods on `IJwtTokenService` | Standardizes safe token storage/validation primitives for upcoming session rotation and replay detection flows without exposing plaintext token material in persistence or diagnostics | 2026-03-15 |
| D010 | JWT bearer auth now performs application-level session checks via `IAccessTokenSessionValidator` and returns a structured `ApiResponse<object>` on unauthorized challenges | Ensures signed tokens are accepted only when backing session + principal token version are still valid while preserving non-leaky unauthorized response contracts for clients/diagnostics | 2026-03-15 |
| D011 | Auth conventions are centralized via shared claim/role constants and refresh-token cookie settings/service, with refresh tokens delivered via HttpOnly secure-ready cookie on admin login | Eliminates drift in role/claim naming across JWT issuance/validation, codifies Guest/Customer/Admin/SuperAdmin policy boundaries, and standardizes frontend-safe refresh token transport without exposing token material in API response bodies or logs | 2026-03-15 |
| D012 | Customer registration upgrades an existing reservation-created customer row when `normalized_email` matches and no password credential exists; otherwise duplicate registration is rejected | Preserves historical guest-customer continuity while preventing account forking or silent ownership takeover for already-credentialed emails | 2026-03-15 |
| D013 | Customer refresh-token rotation uses session replacement semantics (`revoked_at_utc` + `replaced_by_session_id`) instead of cookie-only overwrite | Keeps server-side revocation lineage inspectable and enables replay/failure diagnostics without trusting client cookie state | 2026-03-15 |
| D014 | Customer credential storage is nullable (`customers.password_hash`) and reservation customer resolution must use `normalized_email` lookups | Preserves pre-auth reservation-created customer compatibility while preventing case-drift duplicate identities during reservation-driven customer upsert flows | 2026-03-15 |
| D015 | Customer auth endpoints enforce non-enumerating unauthorized responses for login/register failures while applying 5-attempt/15-minute lockout mutations only on credentialed customer principals | Keeps client-visible auth failures generic (`ApiResponse<object>`) to avoid account-discovery leakage, but preserves actionable lockout diagnostics (`failed_login_count`, `lockout_end_utc`, `last_login_at_utc`) in persisted state and tests | 2026-03-15 |
| D016 | Session validation treats `replaced_by_session_id` as revoked-equivalent, not just `revoked_at_utc` | Hardens replay/lifecycle enforcement against partial or out-of-order session-state writes so rotated-out sessions are denied consistently by access-token validation | 2026-03-15 |
| D017 | Password reset backend contracts are principal-aware (`Customer`/`Admin`) and must be non-enumerating while successful reset bumps `token_version` and revokes active sessions for that principal | Resolves S04 reset-scope ambiguity without cross-principal leakage and guarantees old access/refresh material is invalidated after credential rotation | 2026-03-15 |
| D018 | Admin auth lifecycle now mirrors customer semantics: normalized-email login lockout mutations, refresh-session rotation lineage (`revoked_at_utc` + `replaced_by_session_id`), and logout revocation bound to authenticated `sid`/principal claims | Keeps replay/lockout diagnostics and security guarantees consistent across principal types while preventing cookie-only logout behavior and session-mismatch blind success paths | 2026-03-15 |
| D019 | Password-reset request persists hash-at-rest token rows and invokes an injectable dispatch contract while always returning a non-enumerating success payload (including dispatch-failure cases, logged as warnings) | Prevents account-discovery leakage through delivery-path differences and keeps reset lifecycle/state transitions inspectable for retries and diagnostics without exposing token material | 2026-03-15 |
| D020 | Admin-user management now enforces canonical role values (`Admin`/`SuperAdmin`) through `AuthRoleNames.TryNormalizeAdminRole` and exposes SuperAdmin-only password-reset initiation that reuses hash-at-rest token + dispatcher patterns | Eliminates role-string drift across creation/role updates and keeps admin reset-trigger behavior aligned with the existing secure reset lifecycle/observability contract | 2026-03-15 |
| D021 | Frontend auth integration is implemented as Next.js BFF route handlers (`/api/auth/*`) + `proxy.ts` route protection, with short-lived access JWT in HttpOnly `rac_access` cookie and backend refresh cookie pass-through | Avoids browser CORS/cookie drift, centralizes auth contract translation in one seam, and enables deterministic role-aware route gating/refresh continuity in App Router | 2026-03-15 |
